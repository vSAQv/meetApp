## 1. Текущее состояние проекта

`meet_app` — это серверная часть приложения знакомств на базе **Spring Boot**. Реализованы ключевые доменные сценарии:

- регистрация/логин пользователей
- JWT-аутентификация и проверка доступа по ролям
- email-верификация
- восстановление пароля
- профили пользователей и “базовые анкеты”
- свайпы и создание матчей
- чат внутри матча (WebSocket/STOMP)
- загрузка фото профилей
- базовые сквозные механизмы: централизованная обработка ошибок, валидация DTO, rate limiting

### 1.1 Архитектура
- **Слоистая структура:** `Controller -> Service -> Repository` внутри каждого доменного модуля приложения.
- **DTO как контракты API:** request/response модели вынесены в `org.example.dto.*`, что отделяет API-контракт от JPA-моделей.
- **Валидация и ошибки:** входные данные валидируются через Jakarta Bean Validation, а ошибки унифицируются в `GlobalExceptionHandler` с форматом `ErrorResponse`.
- **Realtime-чат:** обработка сообщений через `ChatWebSocketController` и публикация в `/topic/chat.{matchId}`.
- **Rate limiting:** реализован AOP через аннотацию `@RateLimit` и `RateLimitAspect` (лимит по ключу `ip:path`).

### 1.2 Код и данные
- **JPA-сущности:** пользователь (`UserAccount`), профиль (`Profile`), свайп (`Swipe`), матч (`Match`), сообщение (`Message`), фото (`ProfilePhoto`) связаны напрямую.
- **Хранилище фото:** `PhotoService` сохраняет загруженные файлы в локальный `uploads/photos` и отдает их как ресурс через `WebConfig`.
- **База данных:** по умолчанию используется `H2` в памяти (через `application.yml`).

### 1.3 Процессы разработки
- **Тестирование:** в репозитории не обнаружены автоматические unit/integration тесты (модульные, slice или e2e).
- **CI/CD:** отсутствует настроенный CI (например, GitHub Actions) для сборки и прогонов тестов.

## 2. Области для улучшения

На основе текущего состояния можно выделить следующие направления, которые дадут наибольшую пользу:

| Категория | Область для улучшения | Ожидаемый эффект |
| :--- | :--- | :--- |
| **Качество** | Добавление модульных/интеграционных тестов | Снижение регрессий и уверенность при изменениях логики |
| **Процессы** | Настройка CI (сборка + тесты) | Быстрый feedback и контроль качества |
| **Безопасность** | Отделение JWT секрета от репозитория, добавление политики для токенов | Уменьшение рисков утечки и корректная lifecycle-логика |
| **Интеграции** | Реальная отправка email (а не выдача токена в ответе) | Правильный пользовательский сценарий и сниженный риск утечек |
| **Надежность rate limiting** | Опциональный переход на внешний store (Redis) | Предсказуемость лимитов при масштабировании |
| **Медиа** | Переезд с локального storage на S3/MinIO + ссылки | Лучше для продакшена и масштабирования |
| **Данные** | Продумать схему “включения”/переиспользования профилей, оптимизацию запросов | Ускорение выдачи и снижение нагрузки на БД |

## 3. План развития (устранение технического долга)

Ниже приведен план развития в логической последовательности.

### Этап 1. Укрепление основы (быстрая стабилизация)
1. **Добавить unit/integration тесты для ключевых сценариев**: регистрация/логин, verify-email, восстановление пароля, создание профиля, swipe/match, отправка/получение сообщений.
2. **Включить CI**: сборка проекта и прогон тестов при PR и пушах в основную ветку.
3. **Проверить/усилить security-сценарии**: ограничения `enabled=false`, доступ к сообщениям только участникам матча, корректность rate limiting.

### Этап 2. Безопасность и пользовательские сценарии
4. **Перейти от “токена в ответе” к реальной email-рассылке** (интеграция SMTP/провайдера).
5. **Уточнить жизненный цикл токенов**: одноразовые токены, отработка “usedAt”, настройка TTL и обработка повторных запросов.

### Этап 3. Производительность и масштабирование
6. **Сделать rate limiting масштабируемым** (Redis store), если ожидается горизонтальное масштабирование.
7. **Вынести медиа в object storage** и хранить только URL/ключи.
8. **Оптимизировать БД запросы** (индексы под `userId/matchId`, упорядочивание сообщений, проверка объема выборок).

## 4. Оценка работ

| Мероприятие | Трудоёмкость | Риски |
| :--- | :--- | :--- |
| Тесты ключевых сценариев | Средняя | Возможны выявленные дефекты в текущей логике |
| CI | Низкая | Конфигурационные ошибки, требующие подстройки |
| Усиление security-проверок | Низкая/Средняя | Риск несовместимости со старыми токенами/клиентами |
| Реальная отправка email | Средняя | Зависимость от провайдера, настройка секретов |
| Redis для rate limiting | Средняя | Переоценка сложности и необходимость конфигурации |
| Переезд медиа в object storage | Средняя/Высокая | Требуется миграция и настройка инфраструктуры |

## 5. Сравнение функциональности и технического состояния

### Функциональность
- Основные доменные сценарии реализованы: auth, профили, свайпы/матчи, сообщения, фото.
- Реализованы сквозные функции: DTO+валидация, централизованные ошибки, JWT Security, rate limiting, WebSocket чат.

### Техническое состояние
- Логика работает, но отсутствует “защитный контур” качества: тесты и CI.
- Некоторые сценарии реализованы для разработки упрощенно (например, отдача token в ответе вместо email-рассылки).
- Часть нефункциональных задач требует усиления для продакшена (storage, масштабируемый rate limiting, безопасное управление секретами).

---

## 6. Рекомендации по корректировке плана разработки

Рекомендуется сделать ближайший приоритет смещением в сторону устойчивости и безопасности:

1. Добавить тесты и CI как базу для дальнейших изменений.
2. Довести verify-email и reset-password до продакшн-уровня (реальные email-рассылки).
3. Затем постепенно улучшать производительность и инфраструктуру (Redis для rate limiting, object storage для фото, оптимизация БД).

## Заключение

Технический долг в `meet_app` на текущем этапе связан прежде всего с отсутствием тестов/CI и с “разработческими” упрощениями в интеграционных сценариях (email и хранение медиа). План устранения долга должен начинаться с качества и безопасности, чтобы дальнейшее развитие продукта проходило без регрессий и с предсказуемым результатом.
