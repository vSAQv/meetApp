# UX Improvement Report

## Приложение: `meet_app`

---

# 1. Введение

В данном отчёте представлена оценка удобства использования и предложения по улучшению пользовательского опыта (UX) для системы `meet_app`.

Под “пользователем” здесь понимается:
- разработчик фронтенда/интегратор (в части API),
- пользователь приложения (в части экранов: регистрация, свайп, чат).

Анализ выполнен с опорой на стандарты удобства использования *ISO/IEC 25010:2011 (SQuaRE)* и лучшие практики проектирования API.

Цель улучшений — сделать UX более:
- понятным и логичным,
- предсказуемым для интеграций,
- защищенным от типовых ошибок,
- минималистичным (по духу Badoo: “свайп как основной паттерн”).

---

# 2. Оценка по атрибутам юзабилити ISO 25010

## 2.1 Распознаваемость соответствия

**Текущее состояние (as is):**
- Бэкенд предоставляет REST эндпоинты с понятными ресурсами: `/api/auth/**`, `/api/profiles/**`, `/api/swipes/**`, `/api/messages/**`.
- Для фронтенда важной точкой стала отдача кандидатов для свайпа через `/api/profiles/recommendations`.

**Улучшения (to be):**
- Описать в OpenAPI (Swagger) контракт DTO для всех сценариев UI: verify-email, reset-password, recommendations, swipe, matches, messages.
- Согласовать единые форматы ошибок (`ErrorResponse`) и примеры ответов в документации.

## 2.2 Learnability

**Текущее состояние (as is):**
- В приложении UI используется знакомый пользователю паттерн Badoo: свайп вправо/влево вместо множества кнопок.
- Навигация минимальная: “Свайп / Матчи / Профиль”.

**Улучшения (to be):**
- Добавить подсказки на первом запуске (microcopy): как делать свайп, как перейти в чат.
- В документации к API явно показать последовательность flow: auth -> verify-email -> login -> swipe -> match -> chat.

## 2.3 Используемость

**Текущее состояние (as is):**
- Экран свайпа работает через выборку кандидатов и отправку `SwipeRequest`.
- Экран чата работает через REST (а значит стабильно даже без websocket/handshake-сложностей в среде разработки).

**Улучшения (to be):**
- Рендерить чат более “живым” образом: в будущем заменить polling на WebSocket/STOMP (уже есть серверная реализация).
- Добавить пагинацию/фильтрацию в recommendations (например, `limit`, `cursor`, “исключить уже свайпнутых” на сервере).

## 2.4 Защита от ошибок пользователя

**Текущее состояние (as is):**
- Валидация входных данных реализована на уровне DTO.
- Централизованные ошибки возвращаются единым форматом `ErrorResponse`.

**Улучшения (to be):**
- Для email verification/reset-password добавить более детализированные статусы: “token used”, “token expired”, “user not found”.
- Для свайпа и сообщений — согласовать коды ошибок, чтобы UI мог показывать точные подсказки (например, “Матч не существует”).

## 2.5 Эстетика взаимодействия

**Текущее состояние (as is):**
- UI минималистичный: карточка кандидата + свайповые оверлеи `♥` и `✕`.
- Меньше “лишних” действий на экране: основные действия — свайп/переход в чат.

**Улучшения (to be):**
- Унифицировать отображение фотографий (fallback на initials, затем — один формат под все профили).
- Стандартизировать формат времени (ISO 8601) и единый стиль “меток времени” в сообщениях.

## 2.6 Доступность

**Текущее состояние (as is):**
- UI использует контрастные цвета и крупные элементы (карточка кандидата, зона сообщений).

**Улучшения (to be):**
- Добавить keyboard-friendly сценарии (например, кнопки-подсказки как “скрытые” для screen readers, чтобы свайп можно было эмулировать).
- Уточнить семантику компонентов (aria-label на чат/формы).

# 3. Уровни пользовательского опыта (UX)

## 3.1 Уровень стратегии

Цель системы:
- предоставить удобный UX пользователю знакомств,
- предоставить понятный и интегрируемый API для фронтенда.

## 3.2 Уровень набора возможностей

В UI:
- регистрация/вход,
- verify-email,
- свайп,
- список матчей,
- чат внутри матча.

В API:
- `recommendations` для свайпа,
- `swipe` и `matches`,
- `messages` для истории и отправки.

## 3.3 Уровень структуры

- REST URL: `/api/{module}` с “сквозными” контрактами DTO.
- Стабильная точка входа в чат: `/chat/:matchId`.

## 3.4 Уровень компоновки

- Согласованные структуры payload для:
  - `SwipeRequest`,
  - `MessageResponse`,
  - `CandidateProfileResponse`.

## 3.5 Уровень поверхности

- Минимальные элементы управления (по Badoo):
  - свайп карточки,
  - единая форма сообщения в чате,
  - простая навигация снизу.

# 4. Сравнение “До” и “После”

## Main Screen (Свайп)

### До
- фронтенд вынужден знать `targetUserId` вручную (если нет recommendations).

### После
- UI получает кандидатов через `/api/profiles/recommendations`,
- swipe становится основным UX-паттерном.

## Чат

### До
- без realtime-политик интегратор должен “выдумывать” стратегию обновления.

### После
- чат функционирует по REST (polling), что делает его предсказуемым в dev-среде,
- в будущем возможно переключение на websocket.

# 5. Соответствие рекомендациям

## Microsoft Usability
- Основные действия понятны и соответствуют ожиданиям (свайп/чат).
- Меньше точек принятия решений на экран.

## Google API Design Guide (для интеграторов)
- Понятные ресурсы (`auth`, `profiles`, `swipes`, `messages`).
- Унифицированный формат ошибок.

---

# 6. Планируемые будущие улучшения UX

1. Перейти на websocket для обновлений чата (STOMP `/topic/chat.{matchId}`).
2. Серверная логика рекомендаций: исключение уже свайпнутых на backend.
3. Пагинация/курсор для recommendations и сообщений.
4. Версионирование API (например, `/api/v1/**`) и стабилизация контрактов DTO.

---

# 7. Заключение

Проведённый анализ показал, что `meet_app` имеет хорошую базу для UX: понятные REST-концепции, единый формат ошибок, валидация DTO и минималистичный UI в духе Badoo.

Ключевой практический шаг для улучшения UX — добавление `recommendations`, чтобы свайп был “естественным” для пользователя и не требовал ручного ввода `targetUserId`.

# 9. Заключение по трассируемости

Каждое улучшение привязано к конкретной пользовательской проблеме и к соответствующему атрибуту ISO/IEC 25010. Это обеспечивает системный подход: UX не “косметика”, а измеримая работа над опытом взаимодействия.
