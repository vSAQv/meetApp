# Парное программирование

## 1. Описание выполняемой задачи

В рамках проекта `meet_app` разрабатывалась бэкенд-часть веб-приложения знакомств (аналог Tinder/Badoo). Приложение предоставляет REST API и realtime-чат (WebSocket/STOMP) для сценариев:

- регистрация и вход пользователей,
- верификация email,
- восстановление пароля,
- создание и управление базовыми профилями,
- свайпы и создание матчей,
- обмен сообщениями внутри матчей,
- загрузка фотографий профилей.

Во время работы над задачами использовались два типовых формата парного программирования:

- «на равных»
- «пинг-понг»

## 2. Сравнение применяемых стилей

По итогам совместной работы более удобным оказался стиль **«на равных»**.

В этом формате оба участника активно участвовали в принятии решений: обсуждали контракт API (DTO), границы сервисного слоя, ограничения безопасности (JWT + `@PreAuthorize`), а также способ реализации realtime-части (STOMP-топики и обработчики сообщений). Смена «водителя» происходила после завершения логических блоков (например, после выноса в сервисы `AuthService`, `SwipeService`, `MessageService`), что снижало число расхождений и ускоряло проверку результата.

Стиль **«пинг-понг»** также применялся полезно, но главным образом при работе с более “локальными” задачами: исправление ошибок в обработке исключений (`GlobalExceptionHandler`), настройка rate limiting через AOP (`@RateLimit` + `RateLimitAspect`), а также корректировки синтаксиса схем данных/DDL через JPA.

## 3. Положительные результаты

В процессе работы были получены следующие преимущества:

- **Повышение качества кода.** В паре быстрее выявлялись логические ошибки в проверках прав (участник матча, владение профилем) и валидации входных данных.
- **Обмен опытом.** Лучше закрепились практики Spring Security с JWT, а также реализация WebSocket/STOMP сценариев.
- **Более продуманная архитектура.** Обсуждение flow `REST -> Controller -> Service -> Repository` помогло уменьшить дублирование и сохранить единый подход к DTO.
- **Меньше отвлечений.** При разработке сквозных сценариев (например, сообщение в матче + проверка принадлежности матчу) в паре было проще удерживать контекст.
- **Общее понимание системы.** Ускорилось встраивание новых фич (верификация email/восстановление пароля) за счет синхронизации по моделям и токенам.

## 4. Возникшие трудности и способы их решения

### Разный темп работы
Участники работали с разной скоростью, из-за чего в некоторых задачах возникала пауза.
**Решение:** заранее договариваться о границах “логических” частей кода и фиксировать план обмена ролями по этапам.

### Различия в стиле кодирования
Встречались расхождения в именованиях и организации кода.
**Решение:** договориться о форматировании и использовать единые IDE/линтер-настройки.

### Неочевидные требования к security
В начале могли возникать расхождения по требованиям: какие пользователи должны быть допущены по JWT и как корректно обрабатывать невалидные/неверифицированные аккаунты.
**Решение:** проговаривать критерии “допуска” до реализации и подтверждать их тестовыми сценариями.

## 5. Перспективы применения парного программирования

Парное программирование целесообразно применять:

- при проектировании архитектуры (контракты API, слои приложения);
- при разработке сложной логики авторизации/верификации и валидации;
- при реализации realtime-функций и интеграции WebSocket с доменной моделью;
- при исправлении ошибок и ревью ключевых участков кода (swipe/match/message);
- при рефакторинге, когда важно не потерять поведенческую целостность.

Для простых и рутинных задач индивидуальная работа может быть эффективнее.

## Вывод

Парное программирование для `meet_app` показало себя как полезный инструмент командной разработки: повышает качество, ускоряет обмен знаниями и снижает риск ошибок в сквозных сценариях. При этом для достижения лучшего результата важны согласование формата сессии и четкие критерии “готовности” на каждом этапе.
